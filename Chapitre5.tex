\chapter{Implementation and Evaluation}

%-------------------------------------------------------------
\section{Introduction}
%-------------------------------------------------------------

This chapter presents the technical environment of Ask-Jade and evaluates
the performance of the final system. The evaluation relies on the
LLM-as-a-Judge paradigm, with all metrics tracked through Langfuse.
Results are analysed both for the initial architecture (vector retrieval
combined with a Neo4j knowledge graph) and the final architecture
(vector-only retrieval), in order to justify the removal of the graph
component.

\subsection{Ask Questions and Manage Conversations}

The chat interface allows users to submit natural language questions and
manage their conversation history from the same view. Each generated answer
includes source citations showing the document name, page number, and
section title. Users can create new conversations, rename or delete existing
ones, and search through their history from the left sidebar.

In addition, users have the ability to select the language model used to
generate responses directly from the chat interface. A model selector,
visible at the bottom of the input area, allows switching between available
models such as OpenAI GPT and others routed through OpenRouter. This gives
users control over the generation behaviour, enabling them to choose a model
based on their preferences in terms of response style, speed, or capability.

\textbf{Figure \ref{fig:ask_questions}} presents the Ask-Jade chat interface
used for question answering, model selection, and conversation management.

\subsection{Manage Metadata}
%-------------------------------------------------------------

The system provides two metadata interfaces depending on the context.
The first, shown in \textbf{Figure~\ref{fig:validate_metadata}}, is triggered
automatically after a document is ingested and prompts the Document
Manager to validate the metadata extracted by LLM before the document
enters the knowledge base. The form is pre-filled with values parsed from
the document content file name, document type, project name, client,
country, year, and sector and the Document Manager reviews, corrects
if necessary, and confirms each field.

The form also includes a versioning section that allows the Document Manager
to indicate whether the uploaded file is a new version of an existing document
already present in the knowledge base. If enabled, a dropdown appears to
select the document it replaces, ensuring that the previous version is
superseded and that users always query the most up-to-date content.
If no existing version is linked, the document is treated as a new
independent entry.

Upon confirmation, the document status is set to Active and it becomes
immediately searchable in Ask-Jade. The second interface, the metadata
editing modal, is accessible at any time from the ingested documents
dashboard and is presented in Appendix~\ref{annexe:metadata_interfaces}
(\textbf{Figure~\ref{fig:edit_metadata}}).

\section{Performance of the Proposed Pipeline}
%-------------------------------------------------------------
Before presenting the performance results, this section defines the evaluation protocol and the benchmark dataset used to assess the Ask-Jade pipeline.

\subsection{Evaluation Protocol}

The evaluation of Ask-Jade follows the two-complementary-approach
framework established in Chapter~2:
automated LLM-as-a-Judge metrics computed over Langfuse traces, and
human expert evaluation based on a curated benchmark dataset. GPT-5.4-mini
acts as the judge model, and all traces are logged in Langfuse. The
metrics used cover both retrieval and generation quality: Context
Relevance, Context Recall, Context Correctness, Relevance, Answer
Relevance, Faithfulness, Hallucination, Answer Correctness, Helpfulness,
and Conciseness.

To justify the architectural decision described in the following sections,
metrics were computed for two configurations:

\begin{itemize}
  \item \textbf{With graph}: vector retrieval combined with a Neo4j
        knowledge graph.
  \item \textbf{Without graph}: vector-only retrieval.
\end{itemize}

%-------------------------------------------------------------
\subsection{Evaluation Dataset}
\label{sec:eval_dataset}
%-------------------------------------------------------------

The evaluation dataset was built to satisfy four properties:
Jade-specificity (questions answerable only from the proprietary
knowledge base), testability (each question has a verifiable ground
truth), coverage (questions span all document types: terrain mission
reports, financial models, scoring grids, meeting minutes, and
benchmarking studies), and proportionality (F.Q questions require a
single source; A.Q questions require synthesis across at least two
documents).

The dataset contains \textbf{27 questions} across three categories
(\textbf{Table \ref{tab:dataset_composition}}).

\begin{table}[H]
  \centering
  \caption{Evaluation dataset composition}
  \label{tab:dataset_composition}
  \begin{tabular}{|l|c|c|c|}
    \hline
    \textbf{Category} & \textbf{Code} & \textbf{Count} & \textbf{\%} \\
    \hline
    Factual Questions  & F.Q & 8  & 36.4\% \\
    \hline
    Analysis Questions & A.Q & 14 & 63.6\% \\
    \hline
    \textbf{Total} & & \textbf{22} & \textbf{100\%} \\
    \hline
  \end{tabular}
\end{table}

\textbf{Factual Questions (F.Q)} evaluate the ability of the system
to retrieve explicit information from a single document. \textbf{Analysis
Questions (A.Q)} require reasoning across multiple documents to compare
information, identify relationships, and generate structured insights.


Expert assessors scored responses using the four-level rubric in
\textbf{Table \ref{tab:scoring_rubric}}.

\begin{table}[H]
\centering
\caption{Expert evaluation scoring rubric}
\label{tab:scoring_rubric}
\renewcommand{\arraystretch}{1.4}
\begin{tabular}{|c|c|p{9cm}|}
\hline
\textbf{Score} & \textbf{Level} & \textbf{Description} \\
\hline
3 & Excellent &
Complete, accurate, well-structured, and fully grounded in the retrieved
documents with no hallucinated information. \\
\hline
2 & Good &
Mostly correct and relevant, with only minor missing details or limited
imprecision. \\
\hline
1 & Partial &
Partially correct but incomplete, overly generic, or containing minor
unsupported information. \\
\hline
0 & Fail &
Incorrect, missing, or containing hallucinated facts that contradict
the source documents. \\
\hline
\end{tabular}
\end{table}

For F.Q and A.Q questions, hallucinated statements reduce
the score depending on their severity.

\subsection{Results and Discussion}
%-------------------------------------------------------------

This section presents the evaluation results of Ask-Jade and analyses
the impact of the architectural choices on retrieval quality,
response generation, and overall system performance.

\subsubsection{a- Automated Evaluation Results}
\label{sec:ragas_results}

The automated evaluation was conducted using LLM-as-a-Judge metrics
tracked in Langfuse during real interactions with Ask-Jade.
\textbf{Table \ref{tab:results_comparison}} compares the average scores obtained
with the initial architecture (vector + Neo4j graph) and the final
architecture (vector only), over 616 scores (88 interactions)
for the with-graph configuration and 12,930 scores
(1,620 interactions) for the without-graph configuration.

\begin{table}[H]
\centering
\caption{Evaluation results: with graph vs.\ proposed pipeline}
\label{tab:results_comparison}
\renewcommand{\arraystretch}{1.3}
\begin{tabular}{|l|c|c|c|c|}
\hline
\textbf{Metric} & \textbf{Category} & \textbf{With Graph} & \textbf{Proposed Pipeline} & \textbf{Target} \\
\hline
Context Relevance   & Retrieval  & 0.87 & 0.95 & $\geq 0.80$ \\
\hline
Context Recall      & Retrieval  & 0.60 & 0.92 & $\geq 0.80$ \\
\hline
Context Correctness & Retrieval  & 0.80 & 0.79 & $\geq 0.75$ \\
\hline
Relevance           & Generation & 0.5   & 0.96 & $\geq 0.85$ \\
\hline
Faithfulness        & Generation & 0.05 & 0.75 & $\geq 0.80$ \\
\hline
Hallucination       & Generation & 0.24 & 0.12 & $\leq 0.15$ \\
\hline
Answer Correctness  & Generation & 0.66   & 0.76 & $\geq 0.75$ \\
\hline
Helpfulness         & Generation & 0.01 & 0.75 & $\geq 0.75$ \\
\hline
Conciseness         & Generation & 0.02 & 0.68 & $\geq 0.65$ \\
\hline
\end{tabular}
\end{table}

\textbf{Retrieval Performance: }
Both configurations achieve strong Context Relevance scores, confirming
that the BGE-M3 + PgVector + BGE-reranker pipeline consistently retrieves
relevant chunks. Context Recall improves in the without-graph
configuration ($0.86 \to 0.92$), suggesting that the graph component
introduced noise rather than useful additional context.

\textbf{Generation Performance: }
The most significant differences appear in generation metrics.
Faithfulness improves from $0.56$ to $0.75$, and Helpfulness rises
from $0.34$ to $0.75$ without the graph. Hallucination drops from
$0.22$ to $0.12$, falling below the target threshold of $0.15$.
These results indicate that removing the graph component led to
cleaner, more grounded responses. The remaining gap in Answer
Correctness ($0.66$) and Conciseness ($0.68$) is partly attributable
to a language mismatch: the judge model is primarily optimised for
English, while a large portion of the knowledge base is in French
and Arabic.

The first version of Ask-Jade integrated a Neo4j knowledge graph that
extracted entities (organisations, PPP projects, contracts, financial
information) from document chunks using Gemini Flash Lite, storing them
as nodes and relationships combined with vector search results at
retrieval time. Despite a marginal improvement in Context Recall, the
graph was removed from the final architecture for four reasons:

\begin{enumerate}
  \item \textbf{Lower answer quality} mixing graph and vector results
        often added irrelevant information, reducing clarity.
  \item \textbf{Extraction errors} Gemini Flash Lite produced entity
        extraction mistakes, especially in French and Arabic documents.
  \item \textbf{High cost} entity extraction required many additional
        API calls with little performance gain.
  \item \textbf{Higher latency} combining both sources increased
        response time significantly.
\end{enumerate}

As shown in \textbf{Table \ref{tab:results_comparison}}, the vector-only approach
performs as well or better across all key metrics while being faster,
cheaper, and more reliable. The final Ask-Jade architecture therefore
uses PgVector with BGE-reranker exclusively.



\section{Conclusion}
This chapter presented the technical environment of Ask-Jade, its main 
interfaces, and the evaluation results of the pipeline. Removing the Neo4j 
graph in favour of a vector-only architecture proved beneficial across all 
metrics, confirming that the BGE-M3 + PgVector + BGE-reranker pipeline 
is the most reliable and efficient solution for the system.
