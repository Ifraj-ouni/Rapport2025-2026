\subsection{History}
To understand Large Language Models (LLMs), we must first trace the history 
that shaped them, a journey that, as illustrated in \textbf{Figure 2.2} 
\cite{historyllm}, started slowly during the 1990s and early 2000s, when AI research remained focused on small and less complex projects.

\begin{figure}[H]
    \centering
    \includegraphics[width=0.7\textwidth]{history_ia.jpg}
    \caption{History of LLMs: Complete Timeline and Evolution (1950-2025)}
    \label{fig:historyllm}
\end{figure}


The development of LLMs began in the 1950s with early Natural Language Processing (NLP) research and systems such as machine translation and ELIZA, which demonstrated basic human-computer interaction.

Progress accelerated with the introduction of neural networks and LSTM models in the 1990s, followed by major NLP advances such as Word2Vec and CoreNLP in the 2010s.

A major breakthrough occurred in 2017 with the Transformer architecture, which significantly improved language modeling performance. This led to models such as BERT in 2018, which enhanced contextual understanding of text.

Finally, the release of ChatGPT in 2022 marked a major milestone, making LLMs widely accessible and accelerating their adoption in real-world applications.

\subsection{Limitations of Large Language Models}

Despite their remarkable capabilities, Large Language Models (LLMs) present several limitations that can impact their reliability and effectiveness in real-world applications.

\begin{itemize}

\item \textbf{Hallucinations:} LLMs may generate incorrect or fabricated information that appears convincing, especially when they lack sufficient knowledge about a topic.

\item \textbf{Knowledge Cutoff:} LLMs are trained on static datasets and do not have access to real-time or updated information unless explicitly integrated with external sources.

\item \textbf{Lack of Domain-Specific Accuracy:} Without fine-tuning, LLMs may struggle to provide precise answers in specialized domains such as finance, law, or medicine.

\item \textbf{Limited Context Window:} Although modern models support large contexts, they still cannot process extremely long documents efficiently without losing important information.

\item \textbf{Data Privacy Concerns:} Using sensitive or confidential data with LLMs can raise privacy and compliance issues.

\end{itemize}



These limitations highlight the need for more reliable and context-aware approaches. 
To overcome these challenges, Retrieval-Augmented Generation (RAG) has emerged as an effective solution by combining LLMs with external knowledge sources.




\section{Retrieval Augmented Generation (RAG)}

Retrieval Augmented Generation is a solution to the limitations of LLMs. In this section, we will discuss the definition of RAG, its pipeline, use cases, and the benefits of using RAG in conversational LLMs.

\subsection{Definition}
Retrieval Augmented Generation (RAG) is an architecture to improve an Artificial Intelligence model by connecting it to external knowledge sources. It helps Large Language Models (LLMs) give more relevant and higher quality answers.

\subsection{Rag pipeline}

\textbf{Figure 2.6} shows the RAG pipeline, which retrieves relevant information used by LLMs to generate accurate responses.

\begin{figure}[H]
    \centering
    \includegraphics[width=0.7\textwidth]{RAG_workflow.png}
    \caption{Basic RAG pipeline}
\end{figure}

The RAG pipeline explains how Large Language Models (LLMs) generate accurate answers by using external knowledge sources. The process begins with collecting documents from various sources, forming a comprehensive knowledge base that the system uses to support its responses. Once the data is gathered, it is divided into smaller segments called chunks, allowing efficient processing and search.

Each chunk is then converted into a numerical representation called an embedding, which transforms text into vectors that can be processed by machines. All embeddings are stored in a vector database to enable fast retrieval of the most relevant information.

When a user submits a question, the system converts it into a vector using the same embedding process. It then compares this query vector with stored vectors to retrieve the most relevant chunks. These chunks are combined to form a context that helps the model generate an informed answer.

The LLM receives both the user query and the retrieved context, and by using this external knowledge, it produces responses that are more accurate, coherent, and less prone to hallucinations.

Finally, the generated answer is returned to the user or integrated into a conversational interface, completing the RAG workflow. This ensures that each response is both well-structured and based on relevant and up-to-date information.

\subsection{RAG Use Cases}
RAG techniques are versatile and have been applied in multiple real-world domains, such as: 


\begin{itemize}

\item \textbf{Dynamic and updated information: }RAG allows LLM to give accurate and up-to-date by using real-time information. Without it, a regular LLM would not be different from a RAG-enabled one.

\item \textbf{Reduced Errors: }RAG can make mistakes when they don't have enough knowledge. RAG reduce this by providing relevant information as context, so the LLM can generate more accurate answers.

\item \textbf{Efficient and cost-effective: }RAG indexes data and stores it in a database, so it always has correct information ready.

\item \textbf{Source attributions: }RAG provides links or citations for the data it uses, giving users more context and making the answer more trustworthy.

\item \textbf{More scalable:}RAG can handle large amounts of information and grow with your needs. You can easily add new data or expand to other domains without starting from scratch.

\end{itemize}

\subsection{Embedding Models in RAG Systems}

Embedding models play a crucial role in Retrieval-Augmented Generation (RAG) systems. 
They transform documents and queries into numerical vector representations, enabling semantic search and accurate retrieval. 
Selecting the right embedding model directly impacts \textbf{retrieval accuracy, multilingual support, scalability, and efficiency}.

\subsubsection{Criteria for Choosing an Embedding Model}

When selecting an embedding model for a RAG system, the following criteria should be considered:

\begin{itemize}
    \item \textbf{Multilingual Support:} Ability to handle multiple languages for diverse datasets.
    \item \textbf{Domain Adaptability:} Capacity to represent domain-specific content accurately (e.g., legal, financial, technical).
    \item \textbf{Retrieval Accuracy:} Effectiveness in capturing semantic similarity for search and question-answering tasks.
    \item \textbf{Scalability:} Efficiency for large document collections and high query loads.
    \item \textbf{Embedding Type:} Dense embeddings for semantic similarity; hybrid embeddings for combining dense and sparse representations.
    \item \textbf{Performance \& Efficiency:} Computational cost and speed, especially for real-time or large-scale retrieval.
\end{itemize}
